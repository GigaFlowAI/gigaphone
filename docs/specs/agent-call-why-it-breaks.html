Why agent_call discovery breaks on real OpenHands

All 43 synthetic tests pass — but the implementation-blind tester found two real-world defects. This explains the mechanics of each.

The model in one line

GigaPhone scans each function and asks: does this function dispatch a sub-agent? It says “yes” in two ways:

  • Direct call — the body contains a known agent-SDK call signature (e.g. Runner.run, .invoke, .run).
  • Construct → carrier — the body both constructs a known agent config object (Agent(...), StartConversationRequest(...)) and calls an outbound carrier (.post(...)).

Both checks look at one function at a time, matching on the method name. That is exactly where it breaks.

Defect A  False positives — it matches a method name, not a type

The catalog gave llama-index the signatures calls = (".achat", ".run"). A leading-dot signature matches any call whose dotted name ends in .run:

# match_call_site(dotted)
if sig.startswith("."):
    if dotted.endswith(sig):      # ".run"  →  matches  asyncio.run, subprocess.run, x.run ...
        return sdk                # → tags the enclosing function as agent_call (llama-index)

The matcher has no idea what object .run is called on. asyncio.run, subprocess.run, containers.run all look identical to it — and .run / .invoke / .stream are among the most common method names in Python.

Why the tests missed it: Task 3’s fixture used the qualified two-part signature Runner.run (precise — the receiver Runner is named). No synthetic test had a function that incidentally calls some unrelated .run. The fixture was a clean room; OpenHands is full of incidental .run calls.

Defect B  The real dispatch is missed — the matcher sees one function at a time

The construct→carrier check fires only if one function body contains both a construct and a carrier. OpenHands splits those two steps across a helper call:

The poster owns the .post carrier; the builder owns the StartConversationRequest construct. The matcher walks one function, so it sees only half the signal in each — and a call boundary sits between them.

The fixture put build + post in the same function. Real layered services factor the builder into a helper — so the fixture never exercised the cross-function case.

The common root cause

Both defects are the same shape: the synthetic fixtures were too clean. They used precise qualified names (no incidental .run) and same-function shapes (no factored helpers). Real code is noisy and layered — so the matcher both over-fires (Defect A) and under-fires (Defect B).

Why the precision fix ripples into the fixtures

The natural fix for Defect A is provenance: only match .run if its receiver actually came from the framework’s package. But the test fixtures import fake modules (from wrapper.subagent_sdk import Runner, from agents import Runner), not real framework packages — so a provenance gate keyed on real package names wouldn’t recognize them, and those tests would break unless the fixtures are reworked too. That coupling is why I stopped rather than quietly editing your tests.

The remediation decision

Defect A (false positives) must be fixed regardless — a tool that emits confident wrong answers is worse than one that stays quiet. The open choice is Defect B:

  • Option 2 — fix precision; resolution for B: kill the false positives now, and let the split build/post surface as an unresolved candidate the harness asks about. Less code, honest, but the motivating case isn’t auto-detected.
